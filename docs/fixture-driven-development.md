# Fixture-Driven Development (FxDD) в SotaJS

## Введение: "A-ha" момент

Одна из самых мощных "пасхалок" SotaJS — это возможность разрабатывать и тестировать сложнейшие бизнес-процессы (например, многошаговые воронки в Telegram или флоу оплаты) в полной изоляции от внешнего мира. 

Мы называем это **Fixture-Driven Development (FxDD)**. Благодаря строгой Гексагональной Архитектуре, ваши инфраструктурные адаптеры (БД, API, Мессенджеры) превращаются в "пассивные фикстуры", которые можно мгновенно настраивать и проверять.

## Почему это меняет правила игры?

### 1. Тестирование "Мозга" без "Тела"
Вы можете написать и отладить всю логику Telegram-бота, не имея токена от BotFather и даже не подключаясь к интернету. Вы тестируете не "кнопки", а **состояния и переходы** вашего бизнеса.

### 2. Детерминизм против Хаоса
В реальном мире сложно воспроизвести ситуацию "пользователь нажал кнопку отмены ровно в тот момент, когда повар нажал 'начать готовить'". В FxDD это одна строчка кода в тесте. Вы контролируете время и состояние на 100%.

### 3. Мгновенная обратная связь
Тесты на SotaJS через `bun test` выполняются за миллисекунды. Это позволяет использовать тесты как инструмент "живого проектирования" (Live Design), а не просто как средство проверки перед деплоем.

## Как это работает: Паттерн "Пассивный Адаптер"

В SotaJS каждый адаптер — это просто класс, реализующий порты. Для разработки мы всегда начинаем с `InMemoryAdapter`. В тестах этот адаптер выступает как **Fixture** — стабильный слепок состояния системы.

### Шаг 1: Подготовка адаптера-фикстуры
```typescript
// infrastructure/pizza.adapters.ts
export class InMemoryOrderAdapter implements FeaturePorts<typeof PizzaShopFeature> {
  private orders: Map<string, OrderDto> = new Map();

  async saveOrder(order: OrderDto) {
    this.orders.set(order.id, order);
  }

  // Специальные методы для тестирования (State Inspection)
  clear() { this.orders.clear(); }
  getAll() { return Array.from(this.orders.values()); }
}
```

### Шаг 2: Архитектурный тест-драйв
Вместо запуска бота и ручного кликанья, мы пишем сценарий, который имитирует взаимодействие.

```typescript
// bot.test.ts
describe("Pizza Bot FxDD", () => {
  it("должен блокировать отмену, если заказ уже в печи", async () => {
    // 1. Имитируем команду /pizza через Presentation Layer
    await bot.handleUpdate({ text: "/pizza", chat: { id: 1 }, ... });

    // 2. Имитируем "событие из реальности" — повар начал готовить.
    // Мы манипулируем состоянием напрямую через адаптер-фикстуру.
    const store = new InMemoryOrderAdapter(); 
    const order = (await store.getAll())[0];
    await store.saveOrder({ ...order, status: "cooking" });

    // 3. Пытаемся отменить через интерфейс бота
    await bot.handleUpdate({ text: "/cancel", chat: { id: 1 }, ... });

    // 4. Проверяем состояние фикстуры. 
    // Если бизнес-логика верна, статус останется "cooking".
    const result = (await store.getAll())[0];
    expect(result.status).toBe("cooking"); 
  });
});
```

## Преимущества для разработчика

- **Offline-First**: Работайте в самолете, в поезде, где угодно. Вам не нужны внешние зависимости для создания 100% рабочего "мозга" приложения.
- **Zero Tech Debt**: Вы вынуждены писать чистый код, потому что "грязный" код (завязанный на глобальные переменные или реальные API) просто не пройдет такие тесты.
- **Confidence**: Вы выходите в "продакшен", точно зная, что все краевые случаи обработаны, потому что вы прогнали их через 3D-симулятор вашего бизнеса.
