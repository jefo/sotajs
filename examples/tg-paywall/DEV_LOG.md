# 📓 Журнал разработки: Telegram Paywall Infrastructure
**Проблематика, технические вызовы и инженерные решения.**

Этот документ служит «сырьем» для будущих статей (Habr, VC) и постов в портфолио.

---

## 🛑 Проблема 1: "Localhost Isolation" в мульти-процессной среде
**Ситуация:** Бот (long polling) и Платежный шлюз (HTTP сервер) запущены как разные процессы.
**Боль:** Бот создает подписку в своей базе `paywall.sqlite`, а платежка пытается подтвердить её в своей. Результат — `Subscription not found`.
**Решение:**
- Внедрена общая точка композиции (`paywall.composition.ts`).
- Использование `import.meta.dir` для привязки к абсолютному пути БД в папке примера.
- Теперь процессы работают с единым состоянием (Shared State) независимо от места запуска.

## 🛑 Проблема 2: Ограничения безопасности Telegram (HTTP vs HTTPS)
**Ситуация:** Кнопки запуска Telegram Mini App (TMA) требуют строго HTTPS.
**Боль:** Разработка на `localhost` или `lvh.me` невозможна — Telegram API возвращает 400 Bad Request. Продавать клиенту "сложную настройку туннелей" нельзя.
**Решение:**
- Создан `dev-launcher.ts` — оркестратор среды разработки.
- Интеграция `localtunnel` прямо в рантайм. При старте скрипт сам получает временный HTTPS-адрес и "прокидывает" его через переменные окружения в бот.
- **Инсайт:** Это снижает порог входа для других разработчиков в команду и упрощает демо для заказчика (zero-config).

## 🛑 Проблема 3: Жесткая зависимость от API провайдеров (Vendor Lock-in)
**Ситуация:** У каждого провайдера (Stripe, ЮKassa, Robokassa) свой формат данных, свои подписи (MD5, HMAC) и свои вебхуки.
**Боль:** Фрилансеры обычно пишут логику оплаты прямо в коде бота. Если ЮKassa забанит аккаунт или клиент захочет Stripe — нужно переписывать 50% кода.
**Решение:**
- **Hexagonal Architecture:** Создан порт `PaymentProviderPort` с единым контрактом.
- **Multi-Adapter Strategy:** Реализовано 5+ адаптеров. Смена платежки — это замена одного класса в `Composition Root`.
- **Dynamic Webhook Router:** В `index.ts` написан умный роутер, который по сигнатуре тела запроса (metadata или специфические поля) сам понимает, какой провайдер прислал деньги, и вызывает универсальный Use Case `confirmPaymentCommand`.

## 🛑 Проблема 4: Тестируемость внешних интеграций (Fixture-Driven Development)
**Ситуация:** Нельзя реально оплачивать по 100 раз при разработке, а Mock-объекты в коде не дают "ощущения" реального процесса (UX).
**Боль:** Сложно отлаживать фронтенд TMA и флоу подтверждения без реальной сети.
**Решение:**
- **Network Interceptor:** В режиме `MOCK_NETWORK=true` мы перехватываем `globalThis.fetch`.
- Запросы к API Stripe/ЮKassa не уходят в сеть, а возвращают заготовленный JSON, где `confirmation_url` ведет на наш локальный симулятор шлюза.
- **Результат:** Мы тестируем **реальные адаптеры** с их логикой парсинга, но в изолированной среде.

## 🛑 Проблема 5: Безопасность вебхуков ("Naive Implementation")
**Ситуация:** Кто угодно может узнать URL вебхука и отправить `POST { subscriptionId: "..." }`, получив доступ бесплатно.
**Боль:** Огромные риски для бизнеса эксперта.
**Решение:**
- Внедрен `X-Signature` (HMAC-имитация).
- Шлюз подписывает ID подписки секретным ключом, бот проверяет подпись.
- Это демонстрирует заказчику, что мы думаем о его деньгах на уровне протокола, а не только на уровне "кнопок".
